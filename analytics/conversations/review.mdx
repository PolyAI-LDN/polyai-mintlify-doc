Use **Conversation Review** to see exactly what happened on every voice call and webchat session – the full transcript, the topics the agent matched, the tools it called, and how it handled each turn. It is the primary tool for debugging agent behavior and auditing conversation quality.

![conversations-review](/images/analytics/conv-review.png)

<Note>
For end-to-end QA workflows, see [QA and analytics](/learn/maintain/qa-analytics). For programmatic export, see [Call data](/call-data/introduction).
</Note>

## How the page is organized

The Conversations page has three layers: **channel scopes** at the top, your **saved views** below them, and the **conversations table** that opens an integrated **review side panel** when you select a row.

### Voice and webchat scopes

Conversations are organized by channel. Switch scopes from the tabs at the top of the page:

- **Voice** – calls placed through the voice channel
- **Webchat** – sessions on the [webchat](/webchat/introduction) channel

Each scope has its own default columns (webchat hides voice-only fields like direction, outbound call status, and CSAT) and its own set of saved views, so you can keep voice and webchat workflows completely separate.

### Views

Each scope opens on a built-in **System View** by default:

- **Production** – live conversations only
- **Test** – sandbox, draft, and pre-release conversations

System Views can't be renamed or deleted. To save your own filter and column combinations as pinned tabs, use [Views](/analytics/conversations/views).

### Conversations table

Each row in the table is one conversation. Columns are configurable per view and typically include contact, summary, duration, [PolyScore](/analytics/polyscore), environment, and version.

- **Live calls** appear at the top with a duration indicator and update automatically as new turns arrive.
- **Sort** any sortable column ascending or descending.
- **Select rows** to create a [test case](/analytics/test-suite/introduction) batch from the toolbar.
- **Click a row** to open the conversation in the review side panel without leaving the table.

### Review side panel

Selecting a conversation opens a side panel with four tabs:

| Tab | What you see |
|---|---|
| **Transcription** | Turn-by-turn transcript, audio waveform (voice), matched topics, retrieved sources, intents, user-perceived latency, and the **Diagnosis** toggle group |
| **Scores** | [PolyScore](/analytics/polyscore) and [CSAT](/analytics/csat/introduction) badges with per-dimension breakdowns |
| **Details** | Call info (status, environment, version), pre-built metrics (handoff reason and target, function call, SMS events, A/B group, group ID, QA), and quick links to latency visualization, data logs, and outbound calls |
| **Custom metrics** | Searchable, sortable list of any [custom metrics](/settings/metrics) configured for the project |

Use **Copy Call SID** in the panel header to share a deep link to the conversation. Press **Esc** or click the close icon to dismiss the panel and return to the table.

## Caller identity and PII

The panel header displays the caller's email (webchat) or phone number (voice) when available. Personally identifiable details are visible only to users with [PII-level access](/tools/classes/conv-log#pii). When PII masking is enabled for your account, contact details are redacted in both the table and the panel.

## Matched topics and sources

When the agent matches [Knowledge](/managed-topics/introduction) topics or retrieves content from [Connected Knowledge](/connected-knowledge/introduction) sources, both surfaces appear under the relevant turn:

- **Matched topics** – tags listing the topics most relevant to the user's utterance.
- **Sources** – clickable source names. Clicking a source opens an inline preview panel; use **Open in Knowledge** to jump to the source file.

![sources-conversation-review](/images/release-notes/2603/sources-conversation-review.png)

Enable both layers from the **Diagnosis** toggle group on the Transcription tab. See [Diagnosis](/analytics/conversations/diagnosis) for the full list.

## PolyScore at a glance

Every eligible voice call and webchat session displays a [PolyScore](/analytics/polyscore) badge as a column in the table and on the **Scores** tab – a 0–10 quality rating generated automatically by AI. The badge is color-coded (green for 7–10, amber for 4–6, red for 0–3) and expands to show per-dimension breakdowns. PolyScore works the same way across both channels, so you can run the same QA workflows for voice and chat.

Use PolyScore in Conversation Review to:
- **Prioritize review** – sort or filter the table by PolyScore to surface the lowest-quality calls first.
- **Understand failure modes** – expand the breakdown to see whether a low score reflects poor flow, an unresolved task, or caller frustration.
- **Spot-check high scores** – confirm that well-scored calls followed the right process before using them as benchmarks.

<Tip>
Save a [view](/analytics/conversations/views) filtered by low PolyScore to build a focused QA queue.
</Tip>

## Conversation status

The end of every conversation is visually marked (e.g. **"Ended conversation"**). Live calls show an **"In progress"** badge until they close automatically.

## Default filters

Each scope opens on the **Production** System View. If you are working in Sandbox or have not yet deployed to production, switch to **Test**, or save a Custom View with your preferred filters and **set it as default**.

## Filter by call activity

In addition to metadata and time-based filters, you can narrow the conversations list by what happened during each call:

- **Function calls** – find conversations where the agent invoked a [tool](/tools/introduction). Filter by whether any function ran, or by a specific function name.
- **SMS events** – find conversations where the agent sent an [SMS](/sms/introduction). Filter by whether any SMS was sent, or by a specific SMS event name.

Each filter supports the following operators:

| Operator | Behavior |
|----------|----------|
| **exists** | Match conversations that did (or did not) include the activity. |
| **equals** | Match conversations that include an exact function or SMS event name. |
| **contains** | Match conversations where the function or SMS event name contains the given substring. |

<Tip>
Combine these filters with PolyScore or category tags to focus QA on calls that triggered a specific tool or SMS flow – for example, low-scoring calls where the booking confirmation SMS was sent.
</Tip>

## Data retention

<Warning>
Transcripts older than **90 days** are automatically removed in line with PII retention policies. Plan exports accordingly if you need to keep conversation data for longer.
</Warning>
