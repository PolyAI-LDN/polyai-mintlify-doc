Good agent design is not just about what your agent says. It is about building an **interface between humans and technology using natural language** — and that interface has layers: the conversational copy, the knowledge base, the functions, the state. All of these combine to determine whether a user actually succeeds.

## Task-oriented dialogue

Every agent built in Agent Studio is **task-oriented**. That distinguishes it from open-domain systems designed purely to chat or entertain.

In a task-oriented system, every interaction has a goal. The user wants to book an appointment, check a balance, update a record. The agent's job is to make that happen.

This has two practical consequences:

<CardGroup cols={2}>
  <Card title="You need the right integrations" icon="plug">
    The agent must be connected to the systems that can fulfil the user's need — APIs, functions, state — otherwise it cannot do the thing it was built to do.
  </Card>
  <Card title="Design goes deeper than copy" icon="layer-group">
    The full design surface includes how information flows between the user, the agent, and those internal systems — not just the words the agent speaks.
  </Card>
</CardGroup>

A useful habit when designing a new flow: **start with the data**. What does the API return? What does the function need? What does the user need to see or confirm? The conversational layer is built on top of that foundation.

## The three laws

Every design decision in a task-oriented agent should be judged against three laws, **applied in order**:

<Steps>
  <Step title="Do the thing">
    The user must be able to accomplish their goal. If no path exists to complete the task, nothing else matters.

    This means the right integrations are in place, edge cases have a fallback (channel deflection, live agent handoff), and the agent never leaves a user stranded without a way forward.
  </Step>
  <Step title="Make it easy">
    Once the task is possible, make it as frictionless as possible. This is the core of design work.

    Each step should be obvious. Users should be able to interact in the way natural to them. Unnecessary steps should be removed. The path to completion should be as short as possible.
  </Step>
  <Step title="Add delight">
    Once the interaction works and is easy, polish it. The agent should sound natural and warm, not robotic or terse. Turn-taking, pacing, and tone should all be considered.
  </Step>
</Steps>

<Warning>
  **The order is non-negotiable.** Do not think about delight until the interaction is easy. Do not think about ease until the task is actually possible. If there is ever a conflict between these laws, the earlier one wins.
</Warning>

### Why the order matters

Think of it as a funnel. If some users cannot do the task (law 1 failure), others give up because it is too hard (law 2 failure), and others abandon because the experience is unpleasant (law 3 failure) — you compound those losses. All three laws matter to whether users actually succeed.

A polished agent that cannot complete the task is a failure. An agent that completes the task but is painful to use will lose users before they finish. Engagement is a prerequisite for completion.

<Tip>
  **Law 2 is about making it quick. Law 3 is about making it slick.** Both matter — in that order.
</Tip>

### When the laws conflict: a real example

Collecting a long reference number by voice has a high transcription error rate. Asking the user to type it via DTMF (the phone keypad) is less conversational and adds some friction — it technically degrades the experience.

But if the alternative is failing to capture the number correctly and forcing a handoff after three failed attempts, law 1 takes precedence. Ask for DTMF input.

**Degrading the experience slightly is always better than failing to complete the task.**

## Why sound human

PolyAI agents are designed to sound human. This is a deliberate design choice, not an attempt to deceive.

The reasoning is practical: **humans know how to talk to other humans**. By moving the interface closer to natural conversation — open questions, natural turn-taking, human-like voice — you remove the need for users to learn a new interaction model.

This also means avoiding legacy IVR patterns:

<CardGroup cols={2}>
  <Card title="Avoid" icon="ban">
    **"Press 1 for billing, press 2 for cancellations, press 3 for everything else."**

    This only works for users who fit the options offered, and it is unpleasant for everyone.
  </Card>
  <Card title="Prefer" icon="check">
    **"How can I help you today?"**

    An open question lets users lead with their actual need. Trust the NLU to handle the range of responses.
  </Card>
</CardGroup>

Research shows that even when users know they are speaking to an automated system, they are more comfortable and more successful when the agent behaves in a human-like way. They apply the same conversational heuristics they use with other people — and those heuristics work better when the agent meets them there.

<Info>
  Sounding human is not about tricking people. It is about meeting users where they are — on familiar territory where they already know how to interact.
</Info>

## What this means in practice

When you open Agent Studio and start configuring an agent, keep these questions in mind:

1. **Can every user actually do the thing they came to do?** Are the integrations in place? Are the edge cases handled?
2. **Is the path to completion obvious and short?** Can you remove any steps? Are you asking for information in the right order?
3. **Does this feel like a good experience?** Does the agent sound natural? Is the tone right for the brand and the user?

These are not a checklist to tick once. They are the lens through which every configuration decision should be made.
