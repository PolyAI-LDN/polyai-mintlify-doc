<Info>

## What flows are for

Topics and global rules work well for open-ended conversations. But some interactions need guaranteed sequencing — a registration form, an appointment booking, a multi-step verification. For these, flows let you:

- define each collection step explicitly
- ensure the LLM follows the steps in order and cannot skip ahead
- scope functions to specific steps so the LLM is never shown irrelevant tools
- exit cleanly and hand control back to the general agent

## How flows are structured

A flow consists of one or more **steps**. Each step has a prompt that tells the LLM what to do at that point. The LLM stays in a step until a **flow function** (transition function) is called to advance it.

```
flow: user_registration
  step: ask_phone_number   ← start step
  step: ask_full_name
  step: (exit flow)
```

Each step's prompt appears in the conversation history as a `system` role message, **always appended at the end**. This is what makes flows sticky — the prompt is always the most recent thing the LLM sees, so it cannot be forgotten.

## Creating a flow

You can create a flow from:
- **Build → Flows** → **+** button
- Anywhere you can open the function/topic menu → **+ → Create flow**

When you create a flow, Agent Studio also creates a corresponding **entry function** (e.g. `start_user_registration`) with `conv.go_to_flow("user_registration")` pre-filled.

Every flow needs an entry point. The most common pattern is a Managed Topic whose action calls the entry function:

```
Topic: "User registration"
Sample questions: "I want to register", "Set up a new account"
Action: call start_user_registration
```

Without an entry point, the flow exists but the agent can never enter it.

## Steps and how they work

### The step prompt is always last

When the agent is inside a flow step, the step's prompt is inserted into the conversation history as the **last message** with the `system` role.

```
assistant: "Thanks for calling. How can I help?"
user: "I want to register."
assistant: [calls start_user_registration]
function: (no output)
system: "Ask the user for the phone number we can contact them on."   ← always last
```

No matter how many turns pass, this system message is always the most recent thing the LLM sees. The LLM cannot lose track of the step's instruction.

When the flow transitions to a new step, the **previous step's prompt is removed** from conversation history and replaced with the new step's prompt. The LLM only ever sees the current step — it cannot look ahead or skip.

### Step prompt structure

When a step is active, the LLM sees a system message structured like this:

```
Given the previous context, here are some additional guidelines to help you with your task.

[Your step prompt here]

Make sure to address any question the user asked before proceeding.
Always collect all required pieces of information from the user before proceeding.
Remember to be conversational and brief.
```

The header and footer are added automatically. You only write the middle part.

<Tip>
  The step prompt should tell the LLM exactly what to do at this step and which function to call when done:
  "Ask the user for the phone number we can contact them on. Once you have it, call `save_phone_number`."
</Tip>

## Flow functions (transition functions)

Flow functions are the mechanism for moving between steps. They differ from global functions in two ways:

1. They receive a `flow` object alongside `conv`
2. They use `flow.go_to_step(step_name)` to advance to the next step

```python
def save_phone_number(conv, flow, phone_number: str) -> str:
    """Save the user's phone number and move to the next step."""
    conv.state.phone_number = phone_number
    flow.go_to_step("ask_full_name")
    return None  # no utterance needed — the next step's prompt takes over
```

Flow functions are not listed on the global Functions page. They are scoped to their flow and visible on the flow's **Transitions** panel.

**Key distinction:**
- `conv.go_to_flow(flow_name)` — enters a flow from outside (triggers the start step)
- `flow.go_to_step(step_name)` — moves between steps within the current flow
- `conv.exit_flow()` — exits the flow and returns control to the general agent

## Exiting a flow

Call `conv.exit_flow()` inside any flow function to leave the flow:

```python
def save_full_name(conv, flow, first_name: str, last_name: str) -> str:
    """Save the user's name, complete registration, and exit the flow."""
    conv.state.full_name = f"{first_name} {last_name}"
    # make API call here
    conv.exit_flow()
    return "Registration successful. Your account has been created."
```

After `conv.exit_flow()`:
- The step prompt disappears from conversation history
- The flow functions are no longer visible to the LLM
- The agent returns to normal conversation

## Scoped functions: what the LLM sees in each step

When the agent is inside a step, the LLM can only see functions that are **referenced in that step's prompt**. This is an important difference from global behavior.

Example:
- Step 1 prompt references `save_phone_number` → LLM can only call `save_phone_number`
- After `flow.go_to_step("ask_full_name")` → `save_phone_number` disappears; `save_full_name` becomes available

This means a well-designed flow is also a security and reliability mechanism: the LLM cannot call a function out of sequence because it simply cannot see it at the wrong step.

## What happens when the LLM runs out of instructions

If a step's prompt asks for information but provides no function to call when the information is given, the LLM will improvise. It will draw on its training data to guess what comes next — and this is almost always a hallucination.

**Always give the LLM a way to advance.** Every step should:
1. Tell the LLM what information to collect
2. Reference the function to call once collected

If the LLM has no function to call, it will continue the conversation without constraint.
