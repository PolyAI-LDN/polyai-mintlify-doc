<Info>
  Every function in Agent Studio receives a `conv` parameter automatically — use it to read the current state of the conversation and trigger actions mid-call.
</Info>

## What is the conversation object?

The `conv` object (short for conversation) is passed as the first argument to every LLM-callable function in Agent Studio. You cannot remove it — it is always there.

```python
def my_function(conv, some_parameter: str) -> str:
    # conv is always available here
    return "done"
```

It is a **Python class instance** that gives you two things:

<CardGroup cols={2}>
  <Card title="Properties" icon="database">
    Read-only data about the current call — the caller's number, environment, language, history, and more.
  </Card>
  <Card title="Methods" icon="wrench">
    Actions you can trigger from inside a function — transitioning flows, setting a voice, sending SMS, writing metrics, ending the call.
  </Card>
</CardGroup>

<Tip>
  In Agent Studio's code editor, type `conv.` to see a dropdown of everything available. Properties are marked with a cube icon; methods with a wrench. Hover over each one to see its description and type.
</Tip>

## Properties: data about the call

These are the most commonly used `conv` properties:

| Property | Type | What it gives you |
|---|---|---|
| `conv.account_id` | string | The account identifier for this Agent Studio workspace |
| `conv.project_id` | string | The current project's identifier |
| `conv.id` | string | The unique ID of this conversation (matches Conversation Review) |
| `conv.caller_number` | string | The caller's phone number or chat email address |
| `conv.callee_number` | string | The number the caller dialled |
| `conv.environment` | string | `draft`, `sandbox`, `pre-release`, or `live` |
| `conv.channel_type` | string | `voice` or `chat.polyai` |
| `conv.language` | string | The active language code (e.g. `en-US`) |
| `conv.history` | list | Full conversation history so far |
| `conv.metric_events` | list | All metrics logged in this conversation |
| `conv.variant` | string | The active variant name (if variants are configured) |
| `conv.transcript_alternatives` | list | ASR transcription alternatives for the last user turn |

### Useful patterns with properties

**Gate behaviour by environment** — run test-only logic in sandbox without it reaching live:

```python
if conv.environment in ("draft", "sandbox"):
    return "Test mode: would call API here"
else:
    result = call_real_api()
    return result
```

**Gate behaviour by channel** — send an SMS for voice calls, return a link for chat:

```python
if conv.channel_type == "voice":
    conv.send_sms(to=conv.caller_number, content="Your confirmation: ...")
    return "I've sent you a confirmation by text."
else:
    return "Here is your confirmation link: https://..."
```

## Methods: actions from inside a function

These are the most commonly used `conv` methods:

| Method | What it does |
|---|---|
| `conv.go_to_flow(flow_name)` | Transition to a named flow |
| `conv.exit_flow()` | Exit the current flow |
| `conv.set_voice(voice_config)` | Change the agent's voice mid-call |
| `conv.set_language(language_code)` | Change the active language |
| `conv.set_variant(variant_name)` | Switch to a different variant |
| `conv.write_metric(name, value, write_once=False)` | Log a metric for analytics |
| `conv.send_sms(to, content, ...)` | Send an SMS |
| `conv.call_handoff(destination, reason, utterance)` | Transfer the call |
| `conv.discard_recording()` | Stop recording and prevent persistence |
| `conv.say(utterance)` | Queue a hard-coded utterance (equivalent to returning `{"utterance": "..."}`) |
| `conv.log.info(message, **kwargs)` | Write a structured log visible in Conversation Review and Datadog |

<Tip>
  Prefer `conv` methods over equivalent dictionary return keys where both exist — methods have type hints, validation, and are discoverable from the code editor. If you pass the wrong type, the editor flags it immediately.
</Tip>

### Logging with `conv.log`

Use `conv.log.info(...)` instead of `print()` in production code. Logs appear in Conversation Review under the **Logs** tab, and in Datadog.

```python
conv.log.info("checking environment", environment=conv.environment)
```

`print()` also works and appears in the **stdout** field in Conversation Review — useful for quick debugging, but not recommended for production.

### Writing metrics

Metrics must be defined on the Metrics page before they can be written. Use `write_once=True` when an event should only be counted once per call.

```python
conv.write_metric("booking_attempted", True, write_once=True)
```

## The `conv` object is always up to date

The conversation object is updated continuously throughout the call. `conv.history` grows with each turn. `conv.metric_events` accumulates as metrics are written. Properties like `conv.transcript_alternatives` reflect the most recent ASR output.

This makes `conv` the authoritative source for the current state of the conversation — not the function's local variables, which are discarded as soon as the function returns.
