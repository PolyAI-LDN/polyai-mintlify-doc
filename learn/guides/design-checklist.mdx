Use these 10 rules at three stages:

1. **During design** — as a checklist while you plan
2. **Before build** — have someone else review your design against these rules
3. **After launch** — use them when reviewing live conversations

## The 10 rules

### 1. Guide the user

The agent should make clear how the user can and is expected to respond.

**Explicit guidance** — when you need a specific format, ask for it:

> "I'll need your order number. It should be six digits and starts with a letter — can you go ahead and read that for me?"

**Implicit guidance** — by sounding human, asking open questions, and maintaining natural turn-taking, you set expectations for conversational responses. Users mirror the interaction style you establish.

<Tip>
  Implicit guidance is often more powerful than explicit guidance. If the agent sounds conversational, users give conversational responses — which are richer and easier to work with than one-word answers.
</Tip>

### 2. Listen robustly

The agent should handle the likely range of conversational responses.

If you ask open-ended questions and sound lifelike, you must be able to handle what comes back. Common scenarios:

- **Information in any order** — if you need date, time, and party size for a booking, handle all three at once, one at a time, or any combination
- **Or-questions** — when you ask "Is this international or domestic?", handle: either option, "yes", "no", "both", "neither", "I don't know"
- **Out-of-scope requests** — even if you can't handle mortgages, banking callers will ask about them. Add quick escalation paths for predictable out-of-scope topics

<Warning>
  It is easy to identify out-of-scope topics after launch. It is hard before. Talk to the client, review call samples from the existing contact centre, look at similar projects in the same industry, and use common sense.
</Warning>

### 3. Give feedback

The agent should give timely feedback about what it has understood and what it is doing.

**Implicit confirmation** is usually better than explicit confirmation — it acknowledges the user's intent without adding a turn:

> "To update the address on your account, I'll need your policy number. Do you have that to hand?"

This confirms we heard "update my address" without stopping to ask "Did you say you want to update your address?" If we misunderstood, the user can correct us.

**Show what's happening** — when the agent is doing something behind the scenes (an API call, a lookup), let the user know:

> "Just give me a second to look that up."

### 4. Support correction

The agent should let users undo and correct actions.

- **Exit workflows** — if the user says "No, I don't want to do that", back out cleanly
- **Switch workflows** — if the user says "Actually, I want to cancel", jump directly to cancellation instead of forcing a restart
- **Correct collected values** — if the user says "Sorry, it's five of us, not four", update the value without re-collecting everything

### 5. Prevent errors

The agent should guard against probable and high-impact errors.

Before irreversible actions (booking submissions, payment processing), confirm collected information back to the user:

> "So I've got you down for two people, tomorrow at seven, on the Christmas menu. Should I go ahead and book that?"

This takes priority over acting efficiently. An extra confirmation turn before a non-reversible action is always worth it.

Also: plan for failure. Even if the happy path is smooth, speech recognition may fail, APIs may time out. Stress-test fallback behaviour.

### 6. Act efficiently

Interactions should be quick and lean, especially for frequent tasks.

- Does every step need to be here? Can you combine steps?
- Can agent utterances be shorter?
- If 90% of callers track an order, consider opening with the tracking flow directly instead of "How can I help?"

Efficiency can come at the expense of less frequent tasks. Optimise for the common case.

### 7. Speak clearly and naturally

Speech should be easy to follow with natural pacing.

Err on the side of informality. Think about how a real contact centre agent would say it:

| Avoid | Prefer |
|---|---|
| "Please provide your phone number so we can continue with the booking process." | "And what's your phone number?" |
| "Could you please provide me with your account number?" | "Could you tell me your account number?" |
| "I apologize for the inconvenience." | "Sorry about that." |

If the user is already in a booking flow, you don't need to explain why you need their phone number. They know.

### 8. Behave consistently

The agent should behave and speak in a consistent manner — in voice quality, phrasing, and response length.

**Exception:** A deliberate change in style can signal a change in the interaction. For example, before reading a legal disclaimer, signal the shift: "Okay, I just need to read through some legal bits now." This resets expectations.

Another example: for payment collection via DTMF (keypad input), use a different voice and a brief transfer tone to signal a mode change. Otherwise, users keep speaking to what they think is a conversational agent.

### 9. Be flexible

The agent should support different user needs and preferences.

When offering SMS, always have a backup for users without a mobile phone. Can they go online from a computer? Can you read a URL aloud? Can you send a letter? Can you transfer to a live agent?

Making the task possible for *every* user — not just your ideal user — is essential.

### 10. Adapt to the user

The agent should use what it knows about the user to respond in personalised ways.

If you can see from an API or webhook that the user just cancelled a flight, open with:

> "I can see you just cancelled a flight. Is that what you're calling about?"

This proves competence from the first second, increases engagement, and shortens the interaction.

<Warning>
  Don't overdo personalisation — it can feel intrusive. Use it when it clearly helps the user get to their goal faster.
</Warning>

## Using the checklist

Keep this list visible during design reviews. Not every rule applies to every interaction, but when reviewing a design or a live conversation, ask:

1. Is the user guided clearly?
2. Can the agent handle the range of likely responses?
3. Does the user get feedback about what's happening?
4. Can the user correct mistakes?
5. Are high-impact errors prevented?
6. Is the interaction as efficient as it could be?
7. Does the agent sound natural?
8. Is behaviour consistent (or deliberately varied)?
9. Does the design work for all users?
10. Is available information about the user being used?

<Tip>
  This is not a one-time checklist. Return to it at each stage — design, build, and live review.
</Tip>
