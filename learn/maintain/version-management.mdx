## Quick reference

| I need to... | Action |
|---|---|
| Publish a new version | Click **Publish** → add description |
| Promote to Pre-release | **Deployments > Environments** → Options → Promote to Pre-release |
| Promote to Live | **Deployments > Environments** → Options → Promote to Live |
| Rollback | Find the last good version → Promote to Live |
| Compare versions | Select two versions → **Compare** |
| View history | **Deployments > Environments** |

## Publishing

Publish a version when you've made changes in Sandbox and want to create a checkpoint, test in a clean environment, or prepare for promotion.

<Steps>
  <Step title="Develop in Sandbox">
    Make and test your changes in Sandbox.
  </Step>
  <Step title="Click Publish">
    Click **Publish** in the top right.
  </Step>
  <Step title="Add a description">
    Add a clear version description.
  </Step>
  <Step title="Confirm">
    Click **Publish**.
  </Step>
</Steps>

A published version includes: Managed Topics, Connected Knowledge configs, functions, agent settings, response controls, routing rules, API integrations, and test sets.

<Tip>Use descriptive version descriptions. Examples: "Added Spanish language support with new voice", "Fixed booking function timeout", "Updated holiday hours".</Tip>

<Note>Connected Knowledge content is synced separately. Sync sources before publishing to ensure the latest content is available.</Note>

## Promoting

<Tabs>
  <Tab title="To Pre-release" icon="clipboard-check">
    <Steps>
      <Step title="Open environments">Go to **Deployments > Environments**.</Step>
      <Step title="Select version">Click the **Options Menu** next to the version.</Step>
      <Step title="Promote">Select **Promote to Pre-release**.</Step>
    </Steps>

    Use Pre-release for UAT, validation with select users, and final checks before production.
  </Tab>
  <Tab title="To Live" icon="rocket">
    <Steps>
      <Step title="Open Pre-release tab">Go to the **Pre-release** tab in **Deployments > Environments**.</Step>
      <Step title="Promote">Click Options → **Promote to Live**.</Step>
      <Step title="Confirm">Check the confirmation box and click **Promote**.</Step>
    </Steps>

    <Warning>Promoting to Live affects real customers immediately. Always test in Pre-release first.</Warning>

    You can promote directly from Sandbox to Live, but this is only recommended for emergency hotfixes.
  </Tab>
</Tabs>

## Comparing versions

<Steps>
  <Step title="Open environments">Go to **Deployments > Environments**.</Step>
  <Step title="Select versions">Select two versions and click **Compare**.</Step>
  <Step title="Review the diff">
    - <Icon icon="plus" iconType="solid" color="#2D8C66" /> Green — additions
    - <Icon icon="minus" iconType="solid" color="#D64F4F" /> Red — deletions
    - <Icon icon="rotate" iconType="solid" color="#3B73D1" /> Blue — edits
  </Step>
</Steps>

You can compare Managed Topics, function code, agent settings, response controls, and routing rules.

## Rolling back

Rollback when a new version has critical bugs, performance degrades, or customer complaints spike. Rollback is just promoting an older version:

<Steps>
  <Step title="Open environments">Go to **Deployments > Environments**.</Step>
  <Step title="Find the last good version">Identify the last known good version.</Step>
  <Step title="Promote it to Live">Promote it directly to Live.</Step>
</Steps>

**Time to rollback:** 2-5 minutes. After rolling back, investigate the issue, fix it in Sandbox, and re-promote when ready.

## Deployment workflows

<Tabs>
  <Tab title="Standard" icon="rocket">
    <Steps>
      <Step title="Develop and test in Sandbox">Make and verify your changes.</Step>
      <Step title="Publish with descriptive notes">Create a versioned checkpoint.</Step>
      <Step title="Promote to Pre-release">Validate with UAT testing.</Step>
      <Step title="Promote to Live">Monitor closely after promotion.</Step>
    </Steps>
  </Tab>
  <Tab title="Hotfix" icon="bolt">
    <Steps>
      <Step title="Reproduce the issue">Confirm the bug in Sandbox.</Step>
      <Step title="Fix, test, publish">Apply the fix and create a version.</Step>
      <Step title="Promote through Pre-release">Brief validation, then promote to Live.</Step>
      <Step title="Monitor closely">Watch metrics for the first 30 minutes.</Step>
    </Steps>
  </Tab>
  <Tab title="Seasonal" icon="snowflake">
    <Steps>
      <Step title="Apply seasonal changes">Update hours, policies, etc. in Sandbox.</Step>
      <Step title="Publish and promote">Move through environments for the right date.</Step>
      <Step title="Revert after the season">Rollback or publish new version when the season ends.</Step>
    </Steps>
  </Tab>
</Tabs>

## Best practices

<AccordionGroup>
  <Accordion title="Publish regularly" icon="clock">
    Don't accumulate too many changes in a single version.
  </Accordion>
  <Accordion title="Run test sets before promoting" icon="vial">
    Use [test sets](/analytics/test-suite/introduction) to catch regressions early.
  </Accordion>
  <Accordion title="Monitor after promotion" icon="radar">
    Watch the first 30 minutes closely and check metrics for 24 hours.
  </Accordion>
  <Accordion title="Be ready to rollback" icon="rotate-left">
    Know which version to revert to before promoting.
  </Accordion>
  <Accordion title="Use consistent descriptions" icon="tag">
    Consider tags like `[HOTFIX]`, `[FEATURE]`, `[SEASONAL]`.
  </Accordion>
</AccordionGroup>
