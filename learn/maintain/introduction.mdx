## When to use Maintain vs. Build

| I need to... | Use... | Estimated time |
|--------------|--------|----------------|
| Update existing knowledge (FAQs, hours, etc.) | [Knowledge base](/learn/maintain/knowledge-base) | 5-15 min |
| Sync external sources | [Connected Knowledge](/learn/maintain/knowledge-connected-knowledge) | 10-20 min |
| Change voice or audio | [Voice and audio](/learn/maintain/voice-audio-updates) | 5-10 min |
| Fix a function or update API | [Function maintenance](/learn/maintain/function-maintenance) | 15-30 min |
| Update languages | [Multi-language](/learn/maintain/multi-language-updates) | 20-40 min |
| Investigate performance | [Performance monitoring](/learn/maintain/performance-monitoring) | 15-30 min |
| Deploy or rollback | [Version management](/learn/maintain/version-management) | 5-15 min |
| Add new capabilities | [Build section](/learn/build/introduction) | Hours to days |

## Maintenance workflow

```
1. Identify need → 2. Change in Sandbox → 3. Test → 4. Publish → 
5. Promote to Pre-release → 6. UAT → 7. Promote to Live → 8. Monitor
```

**Typical time:** 30 minutes to 2 hours

## Best practices

- **Test in Sandbox first** - Never make changes directly in Live
- **Use version descriptions** - Document what changed and why
- **Monitor after deployment** - Watch for issues in the first 30 minutes
- **Establish routines** - Daily health checks (10-15 min), weekly reviews (30-60 min)
- **Be ready to rollback** - Know how to revert if something goes wrong

<Tip>
When in doubt, use your [environments](/environments-and-versions/introduction) carefully. Make any changes in the **Sandbox**, test thoroughly, and only then promote up to **Live**.
</Tip>
