Personal access tokens (PATs) let you authenticate to the [Agents API](/api-reference/agents/introduction) as yourself, using your own Agent Studio permissions. Use them when you want to script or automate builder actions — creating agents, editing behavior, publishing deployments — under your own user account, without setting up a separate workspace-scoped API key.

## When to use a PAT

Choose the credential that matches the work you're doing:

| Credential | Use when |
| --- | --- |
| **Personal access token** | You want to call the Agents API as yourself — for local scripts, ad-hoc automation, or coding agents acting on your behalf. Permissions follow your user account. |
| **API key** | You want a long-lived credential for a system-to-system integration (CRM workflows, CCaaS provisioning, internal platforms). Permissions are bound to the key, not a person. See [API keys](/secrets/api-keys). |

PATs inherit your user permissions in real time. If your role changes or you're removed from the workspace, the token's access changes with it.

## Creating a personal access token

1. Open the user menu in the top right of Agent Studio and select **Account**
2. Switch to the **Personal access tokens** tab
3. Select **+ Token**
4. Enter a **Token name** to identify where you'll use it (for example, `local-cli` or `provisioning-script`). The name cannot be changed after creation.
5. Select **Add** to generate the token

<Warning>
Copy your token immediately after creation. The full token value is only shown once — if you lose it, delete the token and create a new one.
</Warning>

You can have up to **2 active personal access tokens** per user account. If you've reached the limit, delete an existing token before creating a new one.

## Viewing and managing tokens

Your tokens appear on the **Personal access tokens** tab. Each entry shows:

- Token name
- Creation date
- A masked token value with a copy button
- A delete button

Deleting a token revokes it immediately. Any scripts or services using it will start receiving `401 Unauthorized` responses on the next request.

## Using your token

Send the token in the `x-api-key` header, exactly like an API key:

```bash
curl https://api.us.poly.ai/v1/agents \
  -H "x-api-key: YOUR_PERSONAL_ACCESS_TOKEN"
```

The Agents API resolves the token to your user account and authorizes the request against your workspace permissions. If you have access to multiple accounts, the account is inferred from the resource (for example, the project ID) in the request path.

## Permissions

Unlike API keys, PATs do not carry their own permissions. Each request is authorized using the **current** permissions of the user that owns the token:

- Adding a user to a new project automatically grants their PATs access to it.
- Removing a user, or reducing their role, immediately reduces what their PATs can do.
- Deactivated users cannot authenticate with their PATs.

This makes PATs well suited to interactive and developer use cases, but a poor fit for shared system integrations — use an API key for those.

## Lifecycle

- Tokens are issued with a fixed expiry. Expired tokens stop working immediately and must be re-created.
- You can delete a token at any time from the **Personal access tokens** tab.
- Tokens are tied to the user that created them. They cannot be transferred to another user — if someone leaves the team, ask the new owner to create their own token.

## Best practices

- Treat tokens as secrets. Store them in your shell's secret manager, a `.env` file ignored by git, or your CI provider's secret store — never in source control.
- Use a descriptive name for each token so you know where it's used when it's time to rotate.
- Rotate tokens periodically by creating a new one, updating the consumers, then deleting the old one.
- Use **API keys** instead of PATs for production system integrations, so credentials don't break when a team member changes role.
