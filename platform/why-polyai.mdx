PolyAI is an enterprise voice AI platform purpose-built for customer service. This page explains the architectural and operational differences that matter when evaluating voice AI platforms for production deployments.

## Proprietary voice AI with Raven

Most voice AI platforms assemble third-party components -- one vendor for speech recognition, another for the LLM, another for text-to-speech. Each handoff adds latency and loses conversational context.

Raven is PolyAI's proprietary LLM family, built as an integrated system of specialized real-time models:

- **Sub-300ms median response latency** -- not from aggressive timeouts, but from eliminating inter-component network hops
- **Conversation-aware endpointing** -- detects true turn endings using linguistic context, not just silence duration
- **Intelligent interruption handling** -- distinguishes real interruptions from backchannel cues ("uh-huh", "right")
- **Noise-resilient processing** -- trained on real call center audio, not clean studio recordings
- **24+ languages** with near-perfect language consistency

See [How Raven works](/agent-settings/how-raven-works) for the full architecture breakdown.

## Structured knowledge architecture

PolyAI's knowledge system is designed for enterprise operations, not just document upload:

### Managed topics

[Managed Topics](/managed-topics/introduction) give you precise control over what the agent says and does. Each topic has sample questions, a curated answer, and optional actions (handoffs, SMS, function calls). This is not a file-upload knowledge base -- it is a structured system with:

- **Weighted retrieval** across topic name, sample questions, and content
- **Action triggering** -- handoffs, SMS, function calls directly from topic matches
- **Flow integration** -- trigger multi-step conversation workflows from any topic
- **Activation control** -- toggle topics per environment for phased rollouts and A/B testing
- **Priority over connected sources** -- Managed Topics always win when there is conflicting information

### Connected knowledge with auto-sync

[Connected Knowledge](/connected-knowledge/introduction) imports external content -- help articles, PDFs, Zendesk, Gladly -- and keeps it synchronized automatically. Unlike file-upload-only models, Connected Knowledge:

- **Auto-syncs** from live sources (Zendesk, Gladly, URLs)
- **Merges at runtime** with Managed Topics for unified retrieval
- **Requires no curation** -- import existing content and let RAG handle matching

### When to use which

| Scenario | Use |
|----------|-----|
| Precise control over agent responses and actions | Managed Topics |
| Large volumes of frequently updated FAQ content | Connected Knowledge |
| Triggering handoffs, SMS, or function calls | Managed Topics |
| Importing existing help center content quickly | Connected Knowledge |
| Both -- they work together | Managed Topics + Connected Knowledge |

## Enterprise deployment pipeline

PolyAI's four-stage deployment pipeline -- Draft, Sandbox, Pre-release, Live -- is designed for enterprise risk management:

- **Draft** -- development workspace for building and iterating
- **Sandbox** -- isolated testing environment with version snapshots
- **Pre-release** -- user acceptance testing (UAT) staging before production
- **Live** -- production, with immediate rollback to any previous version

This structure reduces production incidents by ensuring changes are tested at each stage before reaching live callers. Each environment can have its own phone number, and topic activation is environment-specific.

See [The deployment pipeline](/environments-and-versions/introduction) for the full workflow.

## Testing and quality assurance

PolyAI's [test suite](/analytics/test-suite/introduction) enables regression testing before deployment:

- **Save conversations as test cases** from real interactions or chat testing
- **Group test cases into sets** by feature area (payments, bookings, escalations)
- **Run against Draft or Sandbox** to catch regressions before they reach production
- **Track pass/fail trends** over time to monitor agent quality

Combined with the four-stage deployment pipeline, this creates a safety net where changes are tested against known scenarios before promotion.

## Production monitoring and alerting

PolyAI provides production monitoring through the [Alerts API](/api-reference/alerts/introduction) and [Webhooks API](/api-reference/webhooks/introduction):

- **Real-time metric monitoring** -- track turn latency (p50/p95), API errors, function errors, call crashes, and call volume
- **Configurable alert rules** -- set thresholds and evaluation windows per metric
- **Webhook notifications** -- receive signed HTTP POST requests when alerts trigger or resolve
- **Project-scoped or account-wide** -- monitor individual agents or your entire deployment

See [Production monitoring](/platform/monitoring) for setup and configuration.

## Guided training with PolyAcademy

[PolyAcademy](/learn/guides/polyacademy) is a structured, hands-on training program built into the documentation:

- **Level 1** -- build your first agent from scratch (no code required, ~30 minutes)
- **Level 2** -- add complex topics, connect functions to APIs, fine-tune voice, use diagnostics
- **Level 3** -- code organization, flow patterns, utterance design, and production voice polish

After completing all three levels, you can structure complex projects, design reliable flows, and polish agent speech to production quality.
