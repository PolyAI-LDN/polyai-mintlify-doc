PolyAI is an enterprise voice AI platform purpose-built for customer service. This page explains the architectural and operational differences that matter when evaluating voice AI platforms for production deployments.

## Proprietary voice AI with Raven

Most voice AI platforms assemble third-party components -- one vendor for speech recognition, another for the LLM, another for text-to-speech. Each handoff adds latency and loses conversational context.

[Raven](/agent-settings/raven) is PolyAI's proprietary LLM family -- a coordinated system of specialized real-time models (endpointing, interruption handling, noise filtering, language understanding, response generation) that share context within a single inference pipeline. The result is sub-300ms median response latency across 24+ languages.

See [How Raven works](/agent-settings/how-raven-works) for the full architecture and latency comparison.

## Structured knowledge architecture

PolyAI's knowledge system uses two complementary layers rather than a single flat document store:

- **[Managed Topics](/managed-topics/introduction)** -- curated topics with weighted retrieval, action triggering (handoffs, SMS, function calls), flow integration, and per-environment activation control. See [Why Managed Topics](/managed-topics/introduction#why-managed-topics) for the full comparison with file-upload knowledge bases.
- **[Connected Knowledge](/connected-knowledge/introduction)** -- auto-syncs from live sources (Zendesk, Gladly, URLs) and merges with Managed Topics at runtime. See [Knowledge sources comparison](/connected-knowledge/introduction#knowledge-sources-auto-sync-vs-file-upload) for how this differs from file-upload-only platforms.

Both layers use RAG. Managed Topics always take priority when there is a conflict.

## Enterprise deployment pipeline

PolyAI uses a four-stage deployment pipeline -- Draft, Sandbox, Pre-release, Live -- with per-environment phone numbers, environment-specific topic activation, and instant rollback. See [Why four stages?](/environments-and-versions/introduction#why-four-stages) for the enterprise risk-reduction rationale and the [full deployment workflow](/environments-and-versions/introduction).

## Testing and quality assurance

PolyAI's [test suite](/analytics/test-suite/introduction) lets you save real conversations as test cases, group them into sets by feature area, and run them against Draft or Sandbox after every change. Combined with the four-stage deployment pipeline, this creates a safety net where changes are validated against known scenarios before reaching live callers.

## Production monitoring and alerting

PolyAI provides real-time metric monitoring with configurable alert rules and signed webhook notifications for latency, errors, crashes, and call volume. See [Production monitoring](/platform/monitoring) for the overview and [Alerts API](/api-reference/alerts/introduction) for the full reference.

## Guided training with PolyAcademy

[PolyAcademy](/learn/guides/polyacademy) is a three-level, hands-on training program built into the documentation. It takes you from creating your first agent (no code, ~30 minutes) through production-ready flow patterns and voice polish. See the [PolyAcademy page](/learn/guides/polyacademy) for what you'll learn at each level.

## Voice conversation design

PolyAI's [Voice conversation style guide](/voice/conversation-style-guide) covers the linguistic patterns that make voice agents sound human -- social presence markers, brevity, tone calibration -- backed by [PolyAcademy Level 3](/learn/guides/expert/utterance-design) for deeper training.

## Analytics and conversation intelligence

PolyAI's analytics suite includes [dashboards](/analytics/dashboards/introduction) (standard, safety, custom), [conversation review](/analytics/conversations/introduction) with topic citations and source attribution, [PolyScore](/analytics/polyscore) for automated quality scoring, [Smart Analyst](/smart-analyst/introduction), and [CSAT tracking](/analytics/csat/introduction).
