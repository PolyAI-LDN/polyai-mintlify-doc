Studio Assistant is the AI copilot built into PolyAI Studio. Describe what you want in natural language – *"Add a flow that takes a reservation"*, *"What does my opening-hours topic say?"*, *"Plan a refund flow for me"* – and the assistant reads your project, plans the work, and (when you're ready) makes the changes for you.

You'll find Studio Assistant in the left-hand sidebar of any project, and as the chat panel that opens alongside the builder.

<Note>
Studio Assistant works on top of the same building blocks you use manually – flows, topics, entities, functions, handoffs, and agent settings. Anything it does, you can review, edit, or undo.
</Note>

### What you can do

- **Build** – create flows, topics, entities, functions, integrations, and handoffs from a description
- **Update** – change prompts, steps, conditions, and settings on existing elements
- **Explore** – ask how your current agent is configured ("Which flows transfer to a human?")
- **Plan** – produce a step-by-step implementation plan before any change is made
- **Ask** – get answers about PolyAI Studio concepts, features, and best practices

## Modes

Studio Assistant runs in one of three modes. Switch between them using the mode toggle above the message input.

| Mode | What it does | When to use |
|------|--------------|-------------|
| **Agent** | Plans, then executes changes against your draft (creates and updates flows, topics, entities, etc.) | You're ready to build or change something |
| **Plan** | Analyzes the request and produces a detailed implementation plan – without making any changes | You want to review the approach before anything is written |
| **Ask** | Read-only. Answers questions about your agent and PolyAI Studio concepts | You're exploring or learning, not building |

In **Plan** and **Ask** modes the assistant cannot create, update, or delete anything. If you ask it to make changes in those modes, it will tell you to switch to **Agent** mode.

## How it works

Studio Assistant follows a consistent loop for any non-trivial request:

<Steps>

<Step title="Understands your project">
Before answering or making changes, the assistant reads the relevant parts of your agent – the agent settings (purpose, role, personality), and any flows, topics, entities, or functions it needs to reason about. This is how it gives answers grounded in *your* configuration, not generic advice.
</Step>

<Step title="Plans the work">
For any request that creates or modifies content, Studio Assistant first produces a plan: what already exists, what's missing (e.g. opening hours, prices, policy details), and the proposed steps. You approve or refine the plan before anything is written.

Simple operations – renaming, deleting a single topic, connecting two existing flows – skip planning and run directly.
</Step>

<Step title="Executes via Studio's own tools">
When you confirm, the assistant uses the same APIs Studio uses internally to create or update flows, topics, entities, functions, integrations, and handoffs. Every change lands on your draft, so you can review and publish through your normal [environments and versions](/environments-and-versions/introduction) workflow.
</Step>

<Step title="Explains what it did">
The assistant summarizes what it changed and why, in plain language. You can scroll back through the conversation at any time to see the history of decisions on a project.
</Step>

</Steps>

## What Studio Assistant can work on

Studio Assistant operates on the same building blocks you can edit by hand. For any of these, it can read your current configuration, propose changes, and (in **Agent** mode) write the changes to your draft.

| Building block | Examples of what the assistant can do |
|----------------|----------------------------------------|
| **Flows** | Create new flows; restructure steps, transitions, and conditions; rewrite step prompts |
| **Topics** | Create knowledge-base topics from a description; update wording or scope |
| **Entities** | Define entities and their values; update entity definitions used by flows |
| **Functions** | Wire flows to existing functions; describe and create new function specs |
| **Integrations** | Configure integrations and connect them to flows |
| **Handoffs** | Add transfer-to-human steps with the right conditions and metadata |
| **Agent settings** | Set or update purpose, role, and personality |

For complex requests, the assistant breaks the work down internally and delegates to specialised reasoning steps – for example, a planning step that surveys your project and identifies gaps, or a flows step that handles navigation patterns. You don't need to manage these directly; they show up as plan content and tool calls in the conversation log.

## Conversations and history

- Each conversation is saved to the **History** panel, scoped to the project.
- Open the history panel to switch between past chats, resume an in-progress one, or archive conversations you no longer need.
- The assistant remembers everything in the current chat – follow-ups like *"now do the same for the cancellation flow"* work without repeating context.
- File attachments (e.g. example transcripts, policy docs) can be added to a message and the assistant will read them as part of the request.

## Working effectively with Studio Assistant

The same prompt-quality principles that apply to LLMs generally apply here.

### Provide project context

The assistant always reads your agent settings, but explicit context still helps – especially for ambiguous requests.

Instead of:
> *"Add a flow for refunds."*

Try:
> *"Add a refund flow. We only refund within 30 days of purchase, and only for the original payment method. If the customer asks outside that window, transfer to a human."*

The more business rules you state up front, the less back-and-forth you'll have, and the closer the first draft will be to what you want.

### Use Plan mode for anything non-trivial

For changes that touch multiple flows, restructure logic, or introduce new integrations, switch to **Plan** mode first. You'll get:

- A summary of what already exists and what would change
- A list of business-specific information the assistant still needs from you
- The proposed implementation steps

Review the plan, fill in any gaps, then switch to **Agent** mode and tell the assistant to proceed.

### Ask follow-up questions in Ask mode

Once an agent is built, **Ask** mode is the fastest way to navigate it:

- *"Which flows can transfer to a human, and under what conditions?"*
- *"What entities does the booking flow collect?"*
- *"Where do we use the `customer_id` variable?"*

Because Ask mode is read-only, it's safe to use on production-bound drafts without worrying about accidental edits.

### Iterate in small steps

Studio Assistant performs best when each request maps to a clear unit of work – one flow, one topic, one set of related changes. For larger projects, build incrementally:

1. Plan the scope of the agent at a high level.
2. Add core flows one at a time, testing each before moving on.
3. Layer in edge cases, integrations, and handoffs.

This makes review easier and keeps the assistant's plans focused.

## Example prompts

### Building
- *Create a topic that explains our return policy: 30 days, original payment method, exclusions for final-sale items.*
- *Add a flow that books a table. Collect party size, date, time, and name. Confirm before booking.*
- *Create an entity for delivery status with values: pending, shipped, out for delivery, delivered.*

### Updating
- *Update the opening-hours topic – we're now open until 8pm on weekdays.*
- *Change the booking flow so it asks for a phone number before confirming.*
- *Make the refund flow transfer to a human if the order is older than 30 days.*

### Exploring (Ask mode)
- *What does my agent do today?*
- *Which flows reference the `account_lookup` function?*
- *Summarize the personality and role defined in agent settings.*

### Planning (Plan mode)
- *Plan a flow that handles delivery status lookups, including the case where the order isn't found.*
- *Plan how to add SMS confirmation to my existing booking flow.*

## Limits and what to expect

- **Drafts only.** Studio Assistant edits your draft. It never publishes or promotes between [environments](/environments-and-versions/introduction) on its own.
- **One scoped task at a time.** It works best on focused requests. For very large changes, break them into stages.
- **Business knowledge comes from you.** The assistant won't invent prices, hours, policies, or names – if those are missing, planning will surface them as questions.
- **Always reviewable.** Every change appears in your project just like a manual edit, so you can inspect, tweak, or revert it.
