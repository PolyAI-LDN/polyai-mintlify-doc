PolyPhone is a voice widget that lives on your website. A visitor clicks it, their browser opens, and they're talking to your agent in their own voice. No phone number to dial. No app to install. No queue.

You ship it from **Deployment > Widget** in Agent Studio.

![PolyPhone launcher in the bottom-right of a website](/images/polyphone/hero-launcher.png)

<div className="simplified-only">

## Embed it

The install is one script tag. Generate it from **Deployment > Widget > Embed**, then paste it just before `</body>` on your site:

```html
<script src="https://messaging.poly.ai/widget/YOUR_UNIQUE_CODE.js"></script>
```

That's the whole thing. The launcher shows up bottom-right on any page that loads the script. Styling, content, and agent changes take effect on the next page load. No re-embed needed.

<Tip>
**No website yet?** Use the [shareable page](/widgets/test) instead: a hosted URL on PolyAI's domain that anyone can call without logging in. The fastest way to demo your agent.
</Tip>

For Tag Manager, HTTPS notes, and other deployment details, see [Install on your site](/widgets/install). For styling, language, and consent options, see [Configure](/widgets/configure).

<Note>
**Voice-only on the Open platform.** PolyPhone delivers voice over WebRTC. Webchat and live handoffs are enterprise features.
</Note>

</div>

## What it actually is

A visitor clicks the launcher. Their browser opens a WebRTC connection to PolyAI. Your agent picks up. They talk.

That's the whole experience. No callback. No phone tree. The agent's voice (set under **Configure > Agent voice**) is the same voice your customers would hear on any other channel.

</div>

<div className="full-only">

## PolyPhone in the channel mix

PolyPhone is a voice channel, just like a phone number. The only thing that changes between them is how the audio gets to PolyAI.

| | PolyPhone | Numbers (telephony) | Chat (webchat) | Voice (TTS settings) |
|---|---|---|---|---|
| What customers do | Click a button on your site | Dial a phone number | Type in a widget on your site | (configures how your agent sounds) |
| Modality | Voice | Voice | Text | Shared by all voice channels |
| Where you set it up | Deployment > Widget | Channels > Numbers | Deployment > Widget | Channels > Voice |
| Docs | This section | [Numbers](/telephony/introduction) | [Webchat](/webchat/introduction) | [Voice](/voice/introduction) |

PolyPhone and Numbers are two ways into the same voice agent. They share the agent's spoken [voice](/voice/introduction) (set once, under **Channels > Voice**) and the same conversation logic. Switching a visitor between them doesn't require a separate flow.

</div>

## Two ways to ship

Two routes. Most teams start with the first and graduate to the second.

  <Card title="Website widget" icon="code" href="/widgets/install">

The widget has a consent step before the mic opens, mute and end-call controls, and the usual connecting / in-call / ended / error states. Only one call can be active per visitor across tabs. You can lock it to specific domains, run multiple variants per brand or region, and surface whichever languages your agent speaks.

## Consent and compliance

Privacy and consent controls are off by default. Turn them on under [Widget configuration](/widgets/configure).

<Warning>
You're responsible for meeting transparency, disclosure, and consent requirements in every jurisdiction you operate in. Particularly relevant for EU customers and some US states. Review your obligations carefully.
</Warning>

## Things to know

PolyPhone is a voice entry point, not a handoff layer. It runs the call end-to-end on the same PolyAI agent. Voice-to-chat, chat-to-voice, and live-agent transfer aren't part of PolyPhone today.

It runs in the browser, which means mic permissions, network conditions, and mobile-browser quirks matter. Test in your intended deployment environment before going live.

<div className="full-only">
On enterprise plans, PolyPhone sits alongside your phone-number channel and live-handoff setup; use PolyPhone to capture web visitors while your established hotlines keep handling off-site callers.
</div>

## Prerequisites

Before you create a PolyPhone widget, you'll need:

- An Agent Studio project with a published variant.
- The website domain(s) where the widget will be embedded.
- Access to your website's HTML or your tag manager.
- An **HTTPS** site. Browsers require a secure context for microphone access.

You don't need anything else set up to ship PolyPhone. A voice agent and a website is enough.<span className="full-only"> (Phone and Chat widgets share a single Studio surface, which is why both live under **Deployment > Widget**. Pick **Phone** or **Chat** when you create one.)</span>

## Technical details

<Accordion title="How a call connects">

PolyPhone is loaded by a single `<script>` tag on your site. The script injects an iframe pinned to the bottom-right of the page. When a visitor clicks the launcher, the widget requests microphone access, opens a WebSocket to PolyAI's WebRTC gateway, and starts an Opus 48 kHz audio call with your agent.

Two endpoints are involved:

- **HTTPS** to PolyAI's widget script endpoint, which loads the widget JS.
- **WSS** to the WebRTC gateway, which carries signaling and Opus audio to your agent.

Only one PolyPhone call can be active per browser at a time. The widget enforces this with a cross-tab lock so a visitor with multiple tabs open cannot start parallel calls.

</Accordion>

<div className="full-only">
<Accordion title="PolyPhone vs. telephony, side by side">

| | PolyPhone | Telephony |
|---|---|---|
| Transport | WebRTC over the browser | Phone network (PSTN / SIP) |
| Visitor sees | A widget on your website | A phone number to dial |
| Latency / quality | Opus 48 kHz over WebRTC | Carrier-dependent |
| Setup | Embed a script tag | Provision a number, configure SIP |
| Best for | Web-first journeys, support deflection | Inbound call centers, omnichannel routing |

You can run both: PolyPhone for visitors already on your site, telephony for callers from anywhere else. Both connect to the same agent.

</Accordion>
</div>

<Accordion title="Programmatic control (JavaScript SDK)">

The widget script also exposes a `window.PolyphoneAPI` object that lets your page drive the call programmatically, useful when you want to trigger PolyPhone from a custom button, an in-app event, or an existing chat experience.

```js
// Open the widget and start a call in one step.
window.PolyphoneAPI.connectAndCall();

// Pass context to the agent mid-call.
window.PolyphoneAPI.sendInfo("user_signed_in", { userId: "abc123" });

// React to lifecycle events.
window.PolyphoneAPI.onConnected(() => console.log("Call connected"));
window.PolyphoneAPI.onEnded(() => console.log("Call ended"));

// Remove a handler when you're done.
window.PolyphoneAPI.off("connected", myHandler);
```

Available commands: `connect`, `disconnect`, `startCall`, `endCall`, `connectAndCall`, `mute`, `unmute`, `sendInfo`, `isConnected`, `getCurrentSession`, `destroy`. Event registration uses dedicated methods: `onReady`, `onConnecting`, `onConnected`, `onDisconnected`, `onRegistered`, `onUnregistered`, `onRinging`, `onAnswered`, `onEnded`, `onFailed`, `onMuted`, `onUnmuted`, `onCaptureAudio`, `onAudioEnded`, `onInfoSent`, `onInfoReceived`, `onTranscription`, `onError`. Use `off(eventName, handler)` to remove a registered handler.

</Accordion>

<Accordion title="Browser support">

PolyPhone uses standard WebRTC primitives. Verified browsers:

| Browser | Minimum version |
|---|---|
| Chrome / Chromium | 72 |
| Firefox | 60 |
| Safari (macOS) | 14.1 |
| Safari (iOS) | 16 |
| Edge | 79 |

Visitors on unsupported browsers see a disabled call button with a brief explanation. Mobile-browser and in-app-webview behavior can vary, so test in your intended deployment environment before launch.

</Accordion>
