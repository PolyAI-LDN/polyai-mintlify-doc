Connect PolyAI to Dialpad to automate voice interactions through Dialpad's cloud communications platform. Setup is fully self-serve from Agent Studio — see [CCaaS quick setup](/integrations/voice/quick-setup) for the shared flow.

<Note>
Dialpad uses a SIP trunk provisioned by the Dialpad side. PolyAI does not need any Dialpad credentials — only the phone numbers Dialpad will route to PolyAI per environment.
</Note>

## Prerequisites

- Active Dialpad account with admin access.
- A PolyAI project (you'll see its ID — `PROJECT-XXXXXXXX` — in the Studio URL).
- A SIP trunk configured by the Dialpad SIP-trunk provisioning team to route to PolyAI. They will give you the phone numbers you'll enter in Studio.

## Quick setup

<Steps>
  <Step title="Open Configure > Integrations">
    In Agent Studio (`studio.<region>.poly.ai/<workspace-slug>/<project-id>/`), go to **Configure > Integrations** and click the **Dialpad** card.
  </Step>
  <Step title="Enter the phone numbers per environment">
    Add the numbers Dialpad will route to PolyAI for each environment:

    - **Sandbox** — for development.
    - **Pre-release** — for staging.
    - **Live** — for production.

    Numbers must be E.164 with a country code (e.g. `+441234567890`). The form validates against `^\+[1-9]\d{6,14}$`.
  </Step>
  <Step title="Submit">
    PolyAI creates the integration record and an inbound route per environment. You're then redirected to **Build > Call handoffs** to map outbound transfers back to Dialpad.
  </Step>
</Steps>

## What happens on submit

The Studio form posts to PolyAI's Telephony Gateway:

```
POST /api/v1/{account_id}/{project_id}/dialpad
```

```json
{
  "sandbox": ["+15551110001"],
  "prerelease": ["+15551110002"],
  "live": ["+15551110003"]
}
```

Telephony Gateway creates the `dialpad` integration on your project and an inbound route per environment in PolyAI's connector service. Calls to those numbers, once Dialpad's SIP trunk forwards them, will reach your PolyAI agent.

## How calls flow

1. Caller dials a Dialpad number.
2. Dialpad's SIP trunk routes the call to PolyAI.
3. The PolyAI agent handles the conversation.
4. If the call needs a human, PolyAI sends a SIP REFER back to Dialpad to transfer.

## Configuring handoffs

Dialpad uses **SIP REFER** as the default transfer method (no configurable handoff method).

1. Go to **Build > Call handoffs**.
2. Create handoff destinations (each maps to a Dialpad target — e.g. a queue or extension).
3. Reference those destinations in your agent flow.

See [Call handoffs](/call-handoff/introduction) for full configuration options.

## Editing or removing the integration

- **Update routing**: re-open the Dialpad card in Configure > Integrations to add or remove numbers per environment.
- **Disconnect**: the **Disconnect** button calls `DELETE /api/v1/{account_id}/{project_id}/dialpad`, which removes the integration and tears down its inbound routes.

## Testing

1. Place a test call to a sandbox number.
2. Verify the call reaches your agent.
3. Trigger a handoff in the conversation and confirm Dialpad transfers to the right destination.
4. Review the call in **Analytics > Conversations > Voice**.

## Troubleshooting

| Issue | Likely cause | Fix |
|---|---|---|
| Form rejects a phone number | Not E.164 with country code | Use `+<country><number>` (e.g. `+441234567890`). |
| Calls not reaching PolyAI | Dialpad SIP trunk not routing yet, or wrong number | Confirm with Dialpad's trunk provisioning team and double-check the number was entered in Studio for that environment. |
| Handoffs failing | Destination misconfigured in Call handoffs | Check the destination mapping in **Build > Call handoffs**. |
| "Integration already exists" | One Dialpad integration per project | Use **Edit** in Studio (PATCH) instead of creating a new one. |
